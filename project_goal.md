# 智眸 AI 解题系统：项目说明文档

## 1. 项目概述

### 1.1 项目定位

智眸 AI 解题系统面向中学与高校学习场景，目标是构建一个集拍题识别、模型协同解题、结果验证、同类题推荐、学习习惯分析于一体的智能学习平台。

### 1.2 项目目标

1. 接入国内外主流大模型，基于 OpenCLAW 协议实现统一调用与动态切换。
2. 设计多 Agent 协作与投票机制，提升答案稳定性与可信度。
3. 建立题库缓存与用户习惯数据库，降低重复请求成本并提升响应速度。
4. 提供同类题推荐与分层练习，形成学习闭环。
5. 预留外设接入能力，支持校园或机构硬件扩展。

## 2. 系统架构总览

系统采用三层架构：

- 前端层：题目输入、模型选择、答案展示、同类题练习、个人中心。
- 中部层（核心）：模型调度、多 Agent 协同、数据验证、缓存与数据库。
- 后端层（外设）：设备发现、连接管理、驱动适配、运行状态监控。

## 3. 模块详细设计

### 3.1 前端层

**核心功能**

- 题目输入：支持文字、拍照 OCR、语音转文本。
- 模型选择：展示可用模型与默认推荐策略。
- 答案展示：输出答案、推理过程、置信度与参考来源。
- 同类题推荐：按知识点与难度推荐 1 至 N 道练习题。
- 个人中心：解题记录、学习画像、会员权益与功能解锁状态。

**技术选型**

- Web：React 18 或 Vue 3 + TypeScript + Vite。
- OCR：百度 OCR / 阿里云 OCR（按场景路由）。
- 移动端：Flutter 或 React Native。

### 3.2 中部层（核心）

#### 3.2.1 模型调度层

**职责**

- 管理多模型接入、路由、熔断、重试与回退。
- 通过 OpenCLAW 适配器屏蔽不同厂商 API 差异。
- 维护模型白名单、权重、优先级与成本策略。

**模型分层**

- 国外模型：GPT 系列、Claude 系列、Gemini 系列。
- 国内模型：GLM、文心、通义等。

#### 3.2.2 多 Agent 协作层

协作流程：题目解析 Agent、检索 Agent、求解 Agent、校验 Agent 并行协作，最终进入投票与置信度融合。

#### 3.2.3 数据验证层

针对选择题、填空题、计算题、证明题分别设计标准化解析器，统一答案表达后进行等价性判断。

#### 3.2.4 缓存与数据库层


| 数据表/存储 | 主要字段                                                 | 用途         | 技术选型       |
| ------ | ---------------------------------------------------- | ---------- | ---------- |
| 题库缓存表  | `question_hash`、`answer`、`model_source`、`latency_ms` | 热题复用与快速返回  | Redis      |
| 用户习惯表  | `user_id`、`profile_tags`、`solve_preference`          | 个性化排序与推荐   | MySQL 8.0  |
| 机构题库表  | `org_id`、`private_questions`、`custom_answer`         | 机构私有内容管理   | PostgreSQL |
| 解题记录表  | `trace_id`、`full_chain_log`、`cost_tokens`            | 全链路审计与优化分析 | MongoDB    |


#### 3.2.5 缓存策略

输入题目后计算哈希与语义向量，命中则直接返回缓存答案；未命中则执行完整解题流程，并回写缓存与用户画像数据。

用户习惯功能分级：

- Lv.0：基础解题服务。
- Lv.1：累计解题 50 题后，解锁个性化解题顺序推荐。
- Lv.2：累计解题 200 题后，解锁智眸深度解题模式。
- Lv.3：累计解题 500 题后，解锁完整学习画像与名师点评接口。

## 4. 符号与公式说明

### 4.1 符号列表


| 符号              | 含义                  |
| --------------- | ------------------- |
| $q$             | 输入题目                |
| $a_i$           | 第 $i$ 个 Agent 生成的答案 |
| $w_i$           | 第 $i$ 个 Agent 的权重   |
| $s_i$           | 第 $i$ 个答案的标准化得分     |
| $C$             | 最终置信度               |
| $h(q)$          | 题目哈希值               |
| $\mathbf{e}(q)$ | 题目语义向量              |
| $\tau$          | 相似度阈值               |


### 4.2 核心公式

1. **投票融合得分**

$$
S = \sum_{i=1}^{n} w_i \cdot s_i, \quad \sum_{i=1}^{n} w_i = 1
$$

1. **置信度映射**

$$
C = \sigma(\alpha S + \beta), \quad \sigma(x)=\frac{1}{1+e^{-x}}
$$

1. **缓存命中判定**

$$
\text{Hit}(q)=
\begin{cases}
1, & h(q)=h(q') 
1, & \cos(\mathbf{e}(q), \mathbf{e}(q')) \ge \tau 
0, & \text{otherwise}
\end{cases}
$$

## 6. 关键技术点分析

### 6.1 多 Agent 投票一致性

难点在于不同模型输出风格不一致。解决路径是先标准化答案结构，再进行等价性匹配，最后做权重投票与阈值双判定。

### 6.2 题目相似度判定与缓存命中

难点在于文本存在改写但语义一致。解决路径是哈希精确匹配 + 向量相似度匹配的双通道策略，并叠加题型与知识点标签约束。

### 6.3 用户习惯画像与隐私保护

难点在于学习行为数据敏感。解决路径是本地优先存储、上传前脱敏、传输加密、最小权限访问与审计留痕，并对齐等保 2.0 基本要求。

## 7. 实施里程碑建议

- M1（第 1 至 2 周）：完成前端原型、OCR 接入、基础题目输入链路。
- M2（第 3 至 5 周）：完成模型调度、Agent 协同、基础缓存。
- M3（第 6 至 8 周）：完成用户画像、同类题推荐、后台管理能力。
- M4（第 9 至 10 周）：完成外设接入、性能优化、安全加固与试点验收。
